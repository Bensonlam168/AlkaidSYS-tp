## 1. 项目概述与定位

### 1.1 项目基本信息

**项目名称**：AlkaidSYS（瑶光系统）  
**英文名称**：Alkaid System  
**项目定位**：强大、现代、低代码的企业级 SAAS 系统框架  
**开源协议**：MIT  
**当前版本**：v1.0（设计阶段）  
**文档数量**：67 个设计文档

### 1.2 命名寓意

**瑶光（Alkaid）** 是北斗七星斗柄最末端的恒星，象征着：
- ✨ **开创** - 开创新的企业级 SAAS 框架
- 🔄 **变动** - 灵活的架构设计，适应业务变化
- 🚀 **冒险** - 勇于采用最新技术，突破传统框架限制
- 💡 **创新** - 融合 NIUCLOUD 和 Vue Vben Admin 的优势并创新

### 1.3 核心目标

AlkaidSYS 旨在为企业提供：

1. **快速开发能力** - 低代码设计，提高开发效率 10-40 倍
2. **高性能架构** - 基于 Swoole 协程，支持 10K+ 并发
3. **灵活扩展性** - 插件化架构，易于添加新功能
4. **完善的多租户** - 支持共享数据库、独立数据库和混合模式
5. **现代化前端** - 直接使用 Vue Vben Admin 5.x
6. **多端支持** - 覆盖 Admin、PC、小程序、App、H5 所有终端

### 1.4 目标用户

- **企业级 SAAS 应用开发者** - 主要用户群体
- **应用/插件开发者** - 扩展生态系统
- **企业开发团队** - 快速响应业务需求
- **独立开发者** - 构建商业应用

### 2.2 架构特点说明

#### 第 1 层：客户端层
- **Admin 管理端**：直接使用 Vue Vben Admin 5.x，无需开发
- **PC 客户端**：Vue 3 + Vite + Ant Design Vue
- **移动端**：UniApp 跨平台（微信/支付宝/抖音小程序、iOS/Android App、H5）

#### 第 2 层：API 网关层
- **Nginx**：负载均衡、SSL 终止、静态资源服务
- **Swoole HTTP Server**：高性能协程引擎、路由分发、限流控制

#### 第 3 层：应用层
6 大业务应用 + 1 个低代码管理应用（可选）：
- 电商、OA、CRM、ERP、CMS、AI 应用
- 低代码管理应用提供可视化管理界面

#### 第 4 层：插件层
三类插件系统：
- **通用插件**：跨应用使用（支付、短信、存储、邮件）
- **应用专属插件**：特定应用使用（优惠券、审批流、客户画像）
- **低代码插件**：数据建模、表单设计器、工作流引擎

#### 第 5 层：低代码基础层
低代码核心服务：
- Schema Manager（Schema 管理）
- Collection Manager（数据集合管理）
- Field Type Registry（字段类型注册表）
- Validator Generator（验证器生成器）

#### 第 6 层：核心服务层
框架核心服务：
- 用户服务、租户服务、权限服务
- 应用服务、插件服务

#### 第 7 层：数据层
- **MySQL 8.0+**：主数据库，支持主从复制
- **Redis 6.0+**：缓存 + 队列
- **RabbitMQ**：消息队列

### 2.3 多租户架构

支持三种隔离模式：

| 模式 | 适用场景 | 数据量 | 成本 | 性能 |
|------|---------|--------|------|------|
| **共享数据库** | 中小型租户 | <100万/租户 | 低 | 中 |
| **独立数据库** | 大型租户 | >100万/租户 | 高 | 高 |
| **混合模式** | 灵活组合 | 不限 | 中 | 高 |

双重隔离机制：
- `tenant_id`（租户隔离）
- `site_id`（站点隔离）
- BaseModel 自动隔离

### 2.4 应用和插件两层架构

**创新设计**：明确区分应用和插件

| 特性 | 应用（Application） | 插件（Plugin） |
|------|-------------------|---------------|
| **定义** | 完整的业务功能模块 | 功能扩展和增强 |
| **规模** | 大型 | 小型 |
| **独立性** | 可独立运行 | 依赖应用或框架 |
| **数据库** | 独立的表结构 | 扩展表或新增少量表 |
| **示例** | 电商、OA、CRM | 支付插件、审批流插件 |

## 4. 关键创新点和特色

### 4.1 架构创新

#### 1. Swoole HTTP Server 替代 PHP-FPM

**性能提升 10 倍**：
- QPS：8523 vs 856 (10 倍提升)
- 响应时间 P95：120ms vs 450ms (3.75 倍提升)
- 并发用户：10000+ vs 1000+ (10 倍提升)

**技术优势**：
- ✅ 原生协程支持
- ✅ 内置连接池
- ✅ 常驻内存
- ✅ Swoole Table 共享内存缓存

#### 2. 双重隔离机制

**创新点**：
- tenant_id（租户隔离）
- site_id（站点隔离）
- BaseModel 自动隔离
- 三种隔离模式（共享/独立/混合）

**对比 NIUCLOUD**：
- ❌ NIUCLOUD 仅支持共享数据库模式
- ✅ AlkaidSYS 支持三种模式，灵活性更高

#### 3. 应用和插件两层架构

**创新设计**：
- ✅ 明确区分应用和插件
- ✅ 通用插件和应用专属插件
- ✅ 完整的生命周期管理
- ✅ 强大的钩子系统（Action/Filter/Event Hooks）
- ✅ 钩子优先级控制
- ✅ 插件热更新

**对比 NIUCLOUD**：
- ❌ NIUCLOUD 只有"插件"概念，没有区分应用和插件
- ✅ AlkaidSYS 两层架构更清晰，插件系统更强大

### 4.2 技术创新

#### 1. 直接使用 Vben Admin 5.x

**优势**：
- ✅ 节省 6-8 周开发时间
- ✅ 功能完整（权限控制、状态管理、路由管理）
- ✅ 社区维护，持续更新
- ✅ Monorepo 架构

#### 2. Pinia 状态管理 + AES 加密

**特点**：
- ✅ 比 Vuex 更简洁
- ✅ 完整的 TypeScript 支持
- ✅ AES 加密存储（敏感数据保护）
- ✅ 模块化设计

#### 3. Vite 构建优化

**性能优势**：
- HMR < 100ms（vs Webpack 500ms）
- 构建速度 < 2min（vs Webpack 8min）
- 首屏加载 0.8s（vs 2.5s）
- 包体积 850KB（vs 2.5MB）

## 5. 低代码能力深度分析

### 5.1 低代码能力定位

**核心定位**：开发者工具，而非独立的低代码平台

**设计理念**：
1. **插件化核心引擎** - 低代码能力以插件形式提供
2. **可选管理界面** - 提供可视化界面，但不强制使用
3. **多种使用方式** - 支持 CLI + 界面 + API 三种方式
4. **框架深度集成** - 与 AlkaidSYS 无缝集成
5. **聚焦核心 20%** - 实现核心功能，满足 80% 需求

### 5.2 低代码四层架构

```
低代码应用层（Lowcode Application）
    └── 低代码管理应用（可选安装）
         ↓
低代码插件层（Lowcode Plugins）
    ├── lowcode-data-modeling（数据建模插件）
    ├── lowcode-form-designer（表单设计器插件）
    ├── lowcode-workflow（工作流引擎插件）
    └── lowcode-schema-parser（Schema 解析器插件）
         ↓
低代码基础层（Lowcode Foundation）
    ├── Schema Manager（Schema 管理器）
    ├── Collection Manager（Collection 管理器）
    ├── Field Type Registry（字段类型注册表）
    └── Validator Generator（验证器生成器）
         ↓
核心框架层（Framework Core）
    ├── Container（依赖注入容器）
    ├── Event（事件系统）
    ├── Cache（缓存系统）
    └── Queue（队列系统）
```

### 5.3 核心能力清单

#### 1. 数据建模（Data Modeling）

**功能**：
- ✅ 可视化表设计
- ✅ 15+ 种字段类型

- ✅ 数据迁移和版本管理

## 8. 技术风险与挑战

### 8.1 高风险项

#### 风险 1：工作流引擎复杂度高

**风险等级**：🔴 高  
**影响**：开发难度大，可能延期 2-4 周  
**概率**：中

**应对措施**：
1. ✅ 参考成熟的开源项目（Ingenious、n8n）
2. ✅ 分阶段实现：先基础节点，再高级节点
3. ✅ 提前进行技术预研和 POC 验证
4. ✅ 预留充足的开发时间（6周）

### 8.2 中风险项

#### 风险 2：Swoole 协程调试困难

**风险等级**：🟡 中  
**影响**：开发效率降低 20%  
**概率**：高

**应对措施**：
1. ✅ 使用 Swoole Tracker 进行协程追踪
2. ✅ 加强日志记录，记录协程 ID
3. ✅ 编写详细的单元测试和集成测试
4. ✅ 准备降级方案（同步方式）

#### 风险 3：动态 Schema 性能问题

**风险等级**：🟡 中  
**影响**：响应时间增加 50%  
**概率**：中

**应对措施**：
1. ✅ 使用多级缓存（Redis + Swoole Table）
2. ✅ 优化数据库查询（索引、减少 JOIN）
3. ✅ 进行性能测试（目标 <500ms）
4. ✅ 实现 Schema 预加载和缓存预热

### 8.3 低风险项

#### 风险 4：LogicFlow 社区相对较小

**风险等级**：🟢 低  
**影响**：遇到问题难以解决  
**概率**：低

**应对措施**：
1. ✅ 准备备选方案（AntV X6）
2. ✅ 加强技术预研
3. ✅ 关注 LogicFlow 官方更新

#### 风险 5：ThinkPHP 8.0 版本较新

**风险等级**：🟢 低  
**影响**：可能存在未知 Bug  
**概率**：低

**应对措施**：
1. ✅ 关注官方更新
2. ✅ 及时升级版本
3. ✅ 加强测试覆盖

### 8.4 风险监控机制

**监控指标**：
1. ✅ 开发进度（每周评估）
